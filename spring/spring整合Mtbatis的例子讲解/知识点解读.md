# 一、7.1节我没看懂，由于我不熟悉Mybatis，这里在整合Mybatis的时候涉及到的SqlSessionFactoryBean和MapperScannerConfigurer都是干什么的？为什么需要他们？他们出现在这种用法里是为了讲清楚什么？
你这节卡住很正常，因为 **7.1 其实不是在讲 MyBatis 的查询语法本身**，而是在讲：

**“怎么把原本独立使用的 MyBatis，交给 Spring 来管理。”**

所以这里出现的 `SqlSessionFactoryBean` 和 `MapperScannerConfigurer`，本质上都不是为了让你“会写 SQL”，而是为了完成一件事：

> **把 MyBatis 的核心对象和 Mapper，纳入 Spring 容器。**

我按“先讲结论，再讲作用，再讲为什么需要”的顺序给你拆开。

---

## 一、先说整体：这节到底在干什么

你前面看到的“原始操作代码”大概是这种思路：

1. 读取 `mybatis-config.xml`
2. 手动创建 `SqlSessionFactory`
3. 打开 `SqlSession`
4. 从 `SqlSession` 里拿到 `UserMapper`
5. 调用 `findAll()`

也就是：

```java
InputStream in = Resources.getResourceAsStream("mybatis-config.xml");
SqlSessionFactoryBuilder builder = new SqlSessionFactoryBuilder();
SqlSessionFactory factory = builder.build(in);
SqlSession sqlSession = factory.openSession();
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
List<User> all = mapper.findAll();
```

这是一种 **纯 MyBatis 用法**，特点是：

* `SqlSessionFactory` 你自己创建
* `SqlSession` 你自己打开
* `Mapper` 你自己从 `sqlSession.getMapper(...)` 里拿

而 Spring 整合 MyBatis 的目标是：

* `DataSource` 交给 Spring 管
* `SqlSessionFactory` 交给 Spring 管
* `Mapper` 也交给 Spring 管
* 业务类里直接注入 `UserMapper` 就能用

所以才有了这两个东西：

* `SqlSessionFactoryBean`
* `MapperScannerConfigurer`

---

## 二、SqlSessionFactoryBean 是干什么的

你可以把它理解成一句话：

> **它是 Spring 用来创建 MyBatis 的 `SqlSessionFactory` 的。**

### 1. 为什么叫 `FactoryBean`

Spring 里有一种特殊 Bean，叫 `FactoryBean`，它不是“普通对象本身”，而是“专门负责生产另一个对象”的。

这里：

* 你在 Spring 配置里写的是 `SqlSessionFactoryBean`
* 但它真正帮你产出的，是 **MyBatis 的 `SqlSessionFactory`**

所以它相当于一个“适配器”或者“桥梁”：

* Spring 会管理它
* 它再根据你给的 `DataSource`、MyBatis 配置等，组装出 `SqlSessionFactory`

---

### 2. 它解决了什么问题

在纯 MyBatis 里，你要自己写：

```java
SqlSessionFactoryBuilder builder = new SqlSessionFactoryBuilder();
SqlSessionFactory factory = builder.build(in);
```

但既然现在要整合 Spring，就不希望你在代码里手动 build 了。

Spring 的思路是：

> 你只负责在 xml 里声明依赖关系，真正创建对象这件事交给容器。

所以配置成这样：

```xml
<bean class="org.mybatis.spring.SqlSessionFactoryBean">
    <property name="dataSource" ref="dataSource"/>
</bean>
```

意思就是：

* 用 Spring 创建一个 `SqlSessionFactoryBean`
* 把数据源 `dataSource` 注入进去
* 它再帮你创建真正的 `SqlSessionFactory`

---

### 3. 为什么需要它

因为 **MyBatis 能工作的前提之一，就是必须先有 `SqlSessionFactory`**。

而在 Spring 环境里，你希望：

* 这个工厂不用手写代码创建
* 它依赖的数据库连接池也从 Spring 拿
* 以后事务、数据源、Mapper 都统一托管

所以必须有一个东西，负责把：

* Spring 的 `DataSource`
* MyBatis 的 `SqlSessionFactory`

连起来。

这个东西就是 `SqlSessionFactoryBean`。

---

## 三、MapperScannerConfigurer 是干什么的

你可以把它理解成一句话：

> **它是 Spring 用来批量扫描 Mapper 接口，并把这些接口注册成 Spring Bean 的。**

---

### 1. 先看 Mapper 是什么

比如你有接口：

```java
public interface UserMapper {
    List<User> findAll();
}
```

这个接口本身 **没有实现类**。

在纯 MyBatis 里，你能这么用：

```java
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
```

为什么能拿到实现对象？

因为 MyBatis 会在运行时，给这个接口生成一个 **动态代理对象**。
这个代理对象去执行 `UserMapper.xml` 里的 SQL。

也就是说：

* `UserMapper` 不是普通接口实现类
* 它的对象，是 **MyBatis 动态生成的**

---

### 2. Spring 怎么拿到这些 Mapper

如果没有 `MapperScannerConfigurer`，那 Spring 根本不知道：

* 哪些接口是 Mapper
* 这些接口该怎么变成 Bean
* 业务类里注入 `UserMapper` 时，从哪来

所以就要告诉 Spring：

> 去扫描某个包，把里面的 Mapper 接口都找出来，并注册成 Bean。

这就是：

```xml
<bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
    <property name="basePackage" value="com.wood.mapper"/>
</bean>
```

意思是：

* 扫描 `com.wood.mapper` 这个包
* 发现 `UserMapper` 这种接口
* 为它们创建对应的代理对象
* 再放进 Spring 容器

这样你在业务层就能直接注入：

```java
private UserMapper userMapper;
```

---

### 3. 它解决了什么问题

它解决的是：

> **Mapper 接口怎么从“MyBatis 里才能拿到”变成“Spring 里直接注入就能拿到”。**

没有它时，你得这么干：

```java
SqlSession sqlSession = factory.openSession();
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
```

有了它之后，Spring 启动时已经帮你把 `UserMapper` 准备好了，你直接：

```java
<property name="userMapper" ref="userMapper"/>
```

或者后面学注解时直接 `@Autowired`。

---

## 四、为什么这两个东西要同时出现

因为它们分别负责两件不同的事：

### `SqlSessionFactoryBean`

负责：

* 创建 MyBatis 核心工厂 `SqlSessionFactory`

### `MapperScannerConfigurer`

负责：

* 扫描 Mapper 接口
* 基于前面的 `SqlSessionFactory` 给接口创建代理对象
* 注册到 Spring 容器

你可以理解成这条链路：

**DataSource**
→ 给 `SqlSessionFactoryBean`
→ 生成 `SqlSessionFactory`
→ `MapperScannerConfigurer` 利用它
→ 生成 `UserMapper` 代理对象
→ 交给 Spring
→ `UserServiceImpl` 注入使用

---

## 五、为什么“需要他们”

因为整合的目标变了。

### 原来是纯 MyBatis

你自己控制一切：

* 自己读配置
* 自己创建工厂
* 自己开 session
* 自己取 mapper

### 现在是 Spring + MyBatis

希望 Spring 帮你控制一切：

* 数据源由 Spring 管
* 工厂由 Spring 管
* mapper 由 Spring 管
* service 只关心业务调用

所以就必须要有这两个“整合组件”：

* 一个负责把 MyBatis 的核心工厂交给 Spring
* 一个负责把 MyBatis 的 Mapper 代理对象交给 Spring

---

## 六、它们出现在这种用法里，是为了讲清楚什么

这点最关键。

这节并不是想强调它们的 API 细节，而是想讲清楚：

> **Spring 整合第三方框架时，核心思想是：把第三方框架的重要对象，交给 Spring 容器管理。**

在 MyBatis 这个例子里，最重要的对象有两个层级：

### 第一层：核心基础设施

* `DataSource`
* `SqlSessionFactory`

### 第二层：可供业务直接使用的对象

* `UserMapper`

于是就需要两个整合点：

* `SqlSessionFactoryBean`：把 MyBatis 的工厂纳入 Spring
* `MapperScannerConfigurer`：把 MyBatis 的 Mapper 纳入 Spring

所以它们出现在这里，**主要是为了说明整合思路**，不是为了让你背类名。

---

## 七、你可以把它们记成一个很朴素的理解

### `SqlSessionFactoryBean`

相当于：

> “先把 MyBatis 的发动机装进 Spring。”

### `MapperScannerConfigurer`

相当于：

> “再把所有 Mapper 批量接进 Spring，让业务层能直接用。”

---

## 八、如果没有这两个，会怎么样

### 没有 `SqlSessionFactoryBean`

Spring 容器里没有 MyBatis 的核心工厂，后面 Mapper 也没法正常创建。

### 没有 `MapperScannerConfigurer`

虽然有了 `SqlSessionFactory`，但 Spring 不会自动知道 `UserMapper` 是个需要注册的 Bean。
你还是得自己去 `sqlSession.getMapper(UserMapper.class)`，整合就没整明白。

---

## 九、结合你图里的 Service，再看一遍就通了

图里有这种代码：

```java
public class UserServiceImpl implements UserService {
    private UserMapper userMapper;

    public void setUserMapper(UserMapper userMapper) {
        this.userMapper = userMapper;
    }

    @Override
    public void show() {
        List<User> all = userMapper.findAll();
        for (User user : all) {
            System.out.println(user);
        }
    }
}
```

这里的重点是：

**`UserServiceImpl` 根本没碰 MyBatis 原生 API。**

它不再写：

* `Resources.getResourceAsStream(...)`
* `new SqlSessionFactoryBuilder()`
* `openSession()`
* `getMapper()`

它只是：

* 直接拿到 `userMapper`
* 调用 `findAll()`

这恰恰就是整合成功的标志：

> **业务层不再关心 MyBatis 是怎么创建对象的，只关心使用。**

而能做到这一点，就是靠：

* `SqlSessionFactoryBean`
* `MapperScannerConfigurer`

---

## 十、最后给你一个最简记忆版

你先这样记就够了：

* **SqlSessionFactoryBean**：帮 Spring 创建并管理 MyBatis 的 `SqlSessionFactory`
* **MapperScannerConfigurer**：帮 Spring 扫描并注册 Mapper 接口，让它们能被注入使用

它们存在的意义就是：

> **把原本需要你手动获取的 MyBatis 对象，变成由 Spring 自动管理和注入。**

# 二、Mapper和userMapper分别是干嘛的？
这两个名字很容易混，实际上一个是**类别概念**，一个是**具体对象名**。

## 先说结论

### Mapper

一般指的是：

> **MyBatis 的映射接口这一类东西**

比如：

```java
public interface UserMapper {
    List<User> findAll();
}
```

这里的 `UserMapper` 就属于一个 **Mapper 接口**。

所以“Mapper”更多是在说一种角色、一种类型：

* 用来定义数据库操作方法
* 方法会和 `Mapper.xml` 里的 SQL 对应起来

---

### userMapper

一般指的是：

> **`UserMapper` 这个接口对应的一个对象/变量/Bean 名字**

比如在业务类里：

```java
private UserMapper userMapper;
```

这里：

* `UserMapper` 是类型
* `userMapper` 是变量名

就像：

```java
User user;
```

这里：

* `User` 是类
* `user` 是对象变量名

同理：

```java
UserMapper userMapper;
```

这里：

* `UserMapper` 是 Mapper 接口类型
* `userMapper` 是它的实例引用名

---

## 结合你这节里的语境看

### 1. `Mapper`

说的是“Mapper 机制”或者“Mapper 接口”

比如书里说：

* 编写 Mapper 和 Mapper.xml
* 扫描 Mapper
* 生成 Mapper 对象

这里的 `Mapper` 都是在泛指这类接口，例如：

* `UserMapper`
* `OrderMapper`
* `AccountMapper`

---

### 2. `userMapper`

说的是 Spring 容器里那个具体 Bean，或者 Service 里注入的那个属性

例如：

```xml
<property name="userMapper" ref="userMapper"/>
```

这里前一个 `userMapper` 是 `setUserMapper` 对应的属性名，后一个 `userMapper` 是 Spring 容器里的 Bean 名。

再看代码：

```java
private UserMapper userMapper;
```

这里 `userMapper` 就是成员变量。

---

## 为什么会叫 userMapper

因为接口名通常是大驼峰：

```java
UserMapper
```

而 Spring 默认生成 Bean 名时，常常会把首字母变小写：

```java
userMapper
```

所以你会看到：

* 接口：`UserMapper`
* Bean 名：`userMapper`

---

## 最简单类比

你可以类比成：

```java
Car car = new Car();
```

这里：

* `Car`：类型
* `car`：对象名

对应到 MyBatis：

```java
UserMapper userMapper;
```

这里：

* `UserMapper`：Mapper 接口类型
* `userMapper`：这个 Mapper 的对象名

---

## 在 MyBatis 里它到底负责什么

`UserMapper` 这种 Mapper 接口负责声明数据库操作，比如：

```java
public interface UserMapper {
    List<User> findAll();
    void addUser(User user);
    void deleteById(Integer id);
}
```

然后 `UserMapper.xml` 里写 SQL：

```xml
<select id="findAll" resultType="com.xxx.User">
    select * from user
</select>
```

最后你在业务层里拿到 `userMapper` 这个对象，直接调：

```java
userMapper.findAll();
```

你虽然调用的是 Java 方法，但底层执行的是 XML 里的 SQL。

---

## 一句话记忆

* **Mapper**：泛指 MyBatis 的映射接口这一类东西
* **userMapper**：某个具体 Mapper 的对象名/Bean 名/变量名

你这里真正容易混淆的，通常不是 **Mapper 和 userMapper**，而是这三个：

* `UserMapper`：接口类型
* `userMapper`：接口对应的对象
* `UserMapper.xml`：这个接口对应的 SQL 映射文件
