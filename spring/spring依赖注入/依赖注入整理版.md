# Spring 依赖注入整理版

## 一、什么是依赖注入

依赖注入，英文叫 Dependency Injection，简称 DI。
它是 IoC，也就是控制反转，最常见的实现方式。([钝悟][1])

可以这样理解：

一个对象在工作时，往往需要依赖别的对象。
以前，这个对象自己去创建依赖对象，或者自己去查找依赖对象。
现在改成由 Spring 容器在创建 Bean 的时候，把它需要的依赖直接给它。
这就叫依赖注入。([钝悟][1])

所以，DI 的本质就是一句话：

**对象不自己找依赖，而是由容器把依赖送进来。** ([钝悟][1])

---

## 二、为什么要用依赖注入

依赖注入的核心价值有三个。

### 第一，降低耦合

业务类不需要自己创建依赖对象，也不需要知道依赖对象从哪里来。
这样类和类之间的关系会更松。([钝悟][1])

### 第二，更容易测试

如果一个类依赖接口或者抽象类，那么在单元测试时，就可以很方便地替换成 mock 对象或者 stub 对象。([钝悟][1])

### 第三，更容易扩展和维护

容器负责装配对象之间的关系。
程序员只需要声明“我需要什么”，不需要反复写创建对象和拼装对象的代码。([钝悟][1])

---

## 三、怎么理解依赖注入

网页里提到，理解 DI 可以抓住四个问题。([钝悟][1])

### 1. 谁依赖谁

应用程序依赖 IoC 容器。
因为对象需要的外部资源，通常由容器来提供。([钝悟][1])

### 2. 为什么需要依赖

因为业务对象本身无法独立完成所有功能。
它需要其他对象、资源或者配置数据的支持。([钝悟][1])

### 3. 谁注入谁

是 IoC 容器把依赖注入到应用程序中的对象里。([钝悟][1])

### 4. 注入了什么

注入的是对象需要的外部资源。
包括其他对象、资源、常量数据等。([钝悟][1])

---

## 四、依赖注入有哪些模式

网页把依赖注入分成两类：

### 1. 手动注入模式

手动注入，就是提前通过配置或者编程方式，把注入规则安排好。
常见方式包括：

* XML 配置
* Java 注解配置
* API 配置([钝悟][1])

### 2. 自动注入模式

自动注入，也叫自动装配，英文是 Autowiring。
意思是 Spring 根据容器中的 Bean，自动帮你匹配和注入依赖。([钝悟][1])

自动装配的好处是：

* 可以减少很多显式配置
* 对象关系变化时，配置量也会减少([钝悟][1])

不过这篇资料也特别提醒：

**自动装配有局限性，官方并不推荐无脑依赖它。** ([钝悟][1])

---

## 五、自动装配的几种策略

在基于 XML 的配置里，可以通过 `autowire` 属性指定自动装配模式。常见有四种。([钝悟][1])

### 1. no

默认值，不开启自动装配。
所有依赖都要手动指定。([钝悟][1])

### 2. byName

按名称装配。
Spring 会拿属性名去找同名 Bean，然后注入进去。([钝悟][1])

### 3. byType

按类型装配。
Spring 会按属性类型去找对应的 Bean，然后注入进去。([钝悟][1])

### 4. constructor

这是针对构造器参数的按类型装配。([钝悟][1])

---

## 六、`@Autowired` 注入大致经历什么过程

网页把过程概括成三步：

* 先解析注入相关的元信息
* 再进行依赖查找
* 最后执行依赖注入，比如注入到字段或方法上([钝悟][1])

你面试时可以简单说：

**`@Autowired` 的过程，本质上就是先识别要注入什么，再去容器里找，找到后再赋值进去。**

---

## 七、自动装配有哪些限制

这部分很适合面试回答“自动装配有什么问题”。

### 第一，显式配置会覆盖自动装配

如果你手动指定了依赖，那么手动配置优先。([钝悟][1])

### 第二，简单类型不能直接靠自动装配完成

比如基础类型、字符串、类对象等，不适合直接按普通 Bean 自动装配。([钝悟][1])

### 第三，自动装配不够精准

Spring 会尽量避免“猜错”，因此遇到歧义时会报错，而不是随便选一个。([钝悟][1])

### 第四，同类型多个 Bean 会产生歧义

如果容器里有多个同类型 Bean，而你又只想注入一个，Spring 无法确定到底该选哪个，这时通常会报异常。([钝悟][1])

---

## 八、依赖注入有哪些方式

这是面试高频题。
这篇资料列出了五种注入方式。([钝悟][1])

### 1. 构造器注入

通过构造方法把依赖传进来。([钝悟][1])

它的特点是：

* 依赖在对象创建时就确定
* 适合强依赖
* 有利于保证对象状态完整([钝悟][1])

网页还提到，构造器参数的匹配可以按：

* 类型匹配
* 索引匹配
* 名称匹配([钝悟][1])

#### 面试怎么说

构造器注入适合“必须有”的依赖。
如果没有这个依赖，对象就没法正常工作。
它也更有利于写出不可变对象。

---

### 2. Setter 方法注入

Spring 先创建对象，再调用 setter 方法，把依赖设置进去。([钝悟][1])

它的特点是：

* 更灵活
* 适合可选依赖
* 可以和构造器注入混合使用([钝悟][1])

网页明确建议：

**强制依赖用构造器注入，可选依赖用 Setter 注入。** ([钝悟][1])

---

### 3. 字段注入

直接在成员变量上用注解，比如 `@Autowired`、`@Resource`、`@Inject`。([钝悟][1])

它的优点是：

* 写法最省事
* 代码最短([钝悟][1])

但面试里通常不建议把它说成最佳方案。
因为字段注入虽然方便，但可测试性、可读性、依赖显式程度都不如构造器注入。

---

### 4. 方法注入

在普通方法上加 `@Autowired`、`@Resource`、`@Inject`，也可以完成注入。
网页还把 `@Bean` 放在这一类里讨论。([钝悟][1])

这里你可以这样理解：

方法注入的重点是，**Spring 调用这个方法时，把依赖参数传进去。**

---

### 5. 接口回调注入

也就是实现 Aware 系列接口，比如：

* `BeanFactoryAware`
* `ApplicationContextAware`
* `EnvironmentAware`
* `ResourceLoaderAware`
* `BeanNameAware`
* `MessageSourceAware` 等([钝悟][1])

这种方式本质上是：

**让 Bean 通过回调拿到容器或框架提供的对象。** ([钝悟][1])

不过它对 Spring 框架耦合更强，一般不是首选。

---

## 九、依赖注入怎么选

网页给了一个简单选型结论。([钝悟][1])

### 1. 低依赖，优先构造器注入

也就是依赖不多，而且这些依赖都比较重要时，用构造器更合适。([钝悟][1])

### 2. 多依赖，可以考虑 Setter 注入

依赖比较多时，Setter 方式会更灵活一些。([钝悟][1])

### 3. 想图方便，可以字段注入

但这更多是“方便”，不是“最佳实践”。([钝悟][1])

### 4. 声明类时，可以用方法注入

这里主要是指通过配置方法来组织 Bean 和依赖关系。([钝悟][1])

---

## 十、什么是限定注入

当容器里有多个同类型 Bean 时，Spring 需要进一步确定到底注入哪一个。
这就需要限定注入。([钝悟][1])

常见方式是 `@Qualifier`。
它可以：

* 按 Bean 名称限定
* 按分组限定
* 还可以扩展成自定义限定注解([钝悟][1])

你面试时可以说：

**`@Qualifier` 主要用来解决同类型多个 Bean 的注入歧义问题。**

---

## 十一、什么是延迟注入

延迟注入的意思是：

不是一开始就立即拿到依赖对象，
而是在真正需要使用时再去获取。([钝悟][1])

网页提到两种方式：

* `ObjectFactory`
* `ObjectProvider`，并且推荐 `ObjectProvider`。([钝悟][1])

你可以把它理解成：

**普通注入是“先给我”，延迟注入是“等我用到时再给我”。**

---

## 十二、可以注入哪些数据类型

网页把可注入的数据分成两大类。([钝悟][1])

### 1. 基础类型

包括：

* 基础数据类型
* 数字、字符、布尔、枚举
* `String`、`Object`、`Optional`
* `Locale`、`Charset`、`UUID`
* Spring 里的 `Resource` 等([钝悟][1])

### 2. 集合类型

包括：

* 数组
* `List`
* `Set`
* `Map`
* `Properties`([钝悟][1])

所以，Spring 不只是能注入对象，也能注入集合、常量和资源。

---

## 十三、Spring 依赖注入底层大概怎么处理

这部分更偏源码认知，面试答到这里会显得层次比较高。

网页列出的几个关键点是：

* 入口是 `DefaultListableBeanFactory#resolveDependency`
* 依赖描述由 `DependencyDescriptor` 表示
* 自动注入相关注解主要由 `AutowiredAnnotationBeanPostProcessor` 处理
* 通用注解由 `CommonAnnotationBeanPostProcessor` 处理([钝悟][1])

如果你不想背太细，可以记一句：

**Spring 依赖注入底层是通过 BeanFactory、依赖描述符和后置处理器协同完成的。** ([钝悟][1])

---

## 十四、依赖查找和依赖注入有什么区别

网页最后还对比了依赖查找和依赖注入。([钝悟][1])

### 依赖查找

是对象主动去容器里找依赖。
特点是：

* 主动
* 相对繁琐
* 对容器 API 依赖更明显
* 代码侵入业务逻辑更多([钝悟][1])

### 依赖注入

是容器主动把依赖给对象。
特点是：

* 被动接收
* 更方便
* 对业务代码侵入更低
* 不直接依赖容器 API([钝悟][1])

一句话区分：

**依赖查找是自己去拿，依赖注入是别人送过来。**

# 适合朗读的超精简版本

依赖注入，也就是 DI，是 Spring IoC 最常见的实现方式。
它的核心思想是，对象不自己创建依赖，也不自己查找依赖，而是由 Spring 容器在创建 Bean 时，把依赖对象注入进去。([钝悟][1])

依赖注入的主要好处是降低耦合、方便测试、便于扩展和维护。([钝悟][1])

常见的依赖注入方式有五种：构造器注入、Setter 注入、字段注入、方法注入和接口回调注入。([钝悟][1])

一般来说，强依赖更适合用构造器注入，因为对象在创建时就必须拿到依赖。
可选依赖更适合用 Setter 注入，因为它更灵活。
字段注入虽然写起来方便，但在工程实践中通常不是首选。([钝悟][1])

如果容器中存在多个同类型 Bean，就会出现注入歧义，这时通常需要使用 `@Qualifier` 来限定具体注入哪一个 Bean。([钝悟][1])

此外，Spring 还支持延迟注入，也就是先不立刻拿到依赖，在真正使用时再获取，常见方式是 `ObjectProvider`。([钝悟][1])

从整体上看，依赖查找是对象自己去容器里拿依赖，而依赖注入是容器主动把依赖送给对象。
前者更主动，但侵入性更强；后者更符合 Spring 的编程模型。([钝悟][1])

# 问题：4. 方法注入，网页还把 @Bean 放在这一类里讨论。请问@Bean是把方法返回的对象拿来注入吗？ Spring 依赖注入底层是通过 BeanFactory、依赖描述符和后置处理器协同完成的。这句话怎么理解？原文有解释吗？


# 1. `@Bean` 到底是不是“把方法返回的对象拿来注入”？

**更准确的说法是：`@Bean` 先定义并注册一个 Bean，不是直接表示“注入动作”。**
这篇网页在“方法注入”一节里，确实把 `@Bean` 列进去了，但它在这一页里**只是列出来，没有展开解释**。页面“方法注入”部分只有一个清单：`@Autowired`、`@Resource`、`@Inject`、`@Bean`，并没有进一步说明 `@Bean` 的语义。([钝悟][1])

你可以这样理解：

## `@Bean` 做的事

```java
@Configuration
public class AppConfig {

    @Bean
    public UserService userService() {
        return new UserService();
    }
}
```

这里的意思是：

* Spring 会调用 `userService()` 这个方法；
* 把它的**返回值对象**注册到容器里；
* 这个返回值对象就是一个 Bean。

所以：

* **`@Bean` 的核心作用：注册 Bean**
* **不是直接表示“把返回值注入到别处”**

---

## 那它为什么会被放到“方法注入”里？

因为 `@Bean` 标在**方法上**，而且这个方法本身也可能带参数，例如：

```java
@Bean
public OrderService orderService(UserService userService) {
    return new OrderService(userService);
}
```

这里发生了两件事：

### 第一件事：方法参数被注入

Spring 在调用 `orderService(...)` 时，会先从容器里找 `UserService`，把它作为参数传进来。

### 第二件事：方法返回值被注册

`orderService(...)` 返回的 `OrderService` 会被注册成一个新的 Bean。

所以你问的那句话：

> `@Bean` 是把方法返回的对象拿来注入吗？

更严谨地回答是：

**不是“拿来注入”，而是“拿来注册成 Bean”；至于这个 Bean 以后会不会被注入到别的地方，那是后续依赖注入阶段的事。**
这一页原文没有把这个区别讲细，只是把 `@Bean` 归到了“方法注入”这一类。([钝悟][1])

---

# 2. “Spring 依赖注入底层是通过 BeanFactory、依赖描述符和后置处理器协同完成的”这句话怎么理解？

这句话是在对网页“依赖处理过程”那一节做**概括性转述**。因为原文那一节本身写得很简，只列了几个关键类名，没有展开成完整流程。网页原文列出的关键点包括：

* 入口：`DefaultListableBeanFactory#resolveDependency`
* 依赖描述符：`DependencyDescriptor`
* 自定义绑定候选对象处理器：`AutowireCandidateResolver`
* `@Autowired`、`@Value`、`@Inject` 处理器：`AutowiredAnnotationBeanPostProcessor`
* 通用注解处理器：`CommonAnnotationBeanPostProcessor` ([钝悟][1])

也就是说，**原文有这些名词，但基本没有解释“它们怎么协同”。** ([钝悟][1])

---

## 你可以把这句话拆成三层理解

### 一，BeanFactory：真正“找依赖”的核心容器

网页说依赖处理入口是 `DefaultListableBeanFactory#resolveDependency`。这说明当 Spring 真正要解析“这里应该注入什么对象”时，会进入 BeanFactory 的依赖解析逻辑。([钝悟][1])

通俗说：

> **BeanFactory 像仓库管理员。**
> 别人提出“我要一个 `UserService`”，它负责去容器里找合适的 Bean。

---

### 二，DependencyDescriptor：把“我要什么依赖”描述清楚

网页列出了 `DependencyDescriptor`。它可以理解成：

> **依赖请求说明书**

它描述的信息大致是：

* 这是字段注入，还是方法参数注入，还是构造器参数注入；
* 需要的类型是什么；
* 这个依赖是不是必须的；
* 有没有限定条件，比如限定名、泛型信息等。

虽然这篇文章没有展开这些细节，但它把 `DependencyDescriptor` 明确列为“依赖描述符”。([钝悟][1])

---

### 三，BeanPostProcessor：先识别哪里要注入，再触发注入

网页列出了两个后置处理器：

* `AutowiredAnnotationBeanPostProcessor`：处理 `@Autowired`、`@Value`、`@Inject`
* `CommonAnnotationBeanPostProcessor`：处理一些通用注解，例如 `@Resource` 等 ([钝悟][1])

这可以理解为：

> **后置处理器像扫描员。**
> 它先扫描 Bean 的字段、方法、构造器参数，看哪里写了注入注解。
> 发现后，再把这些注入点交给 BeanFactory 去解析和填充。

---

# 3. 串起来看，完整过程是什么？

你可以记成这个顺序：

## 第一步：后置处理器先发现“哪里需要注入”

例如发现：

```java
@Autowired
private UserService userService;
```

或者：

```java
@Autowired
public void setUserService(UserService userService) { ... }
```

这一步主要对应网页里说的注解处理器。([钝悟][1])

---

## 第二步：把这个注入点包装成“依赖描述”

也就是用 `DependencyDescriptor` 表示：

* 需要一个 `UserService`
* 注入位置是字段或方法参数
* 是否必须注入

这一步对应网页里的“依赖描述符”。([钝悟][1])

---

## 第三步：交给 BeanFactory 去解析

BeanFactory 通过 `resolveDependency` 去容器里找匹配的 Bean。
如果只有一个合适的 Bean，就注入它；如果有多个，可能要结合限定条件继续判断；如果没有，就可能报错。网页把这个入口明确写成了 `DefaultListableBeanFactory#resolveDependency`。([钝悟][1])

---

## 第四步：把找到的对象真正设置进去

也就是网页前面概括的三步：

* 元信息解析
* 依赖查找
* 依赖注入（字段、方法） ([钝悟][1])

所以我之前那句“通过 BeanFactory、依赖描述符和后置处理器协同完成”，其实就是把网页这两部分内容合在一起说了：
一部分是“`@Autowired` 注入过程”的三步概括，另一部分是“依赖处理过程”列出的关键类。([钝悟][1])

# 5. 面试里怎么答最稳

你可以直接背这段：

> `@Bean` 的本质不是注入，而是通过方法把返回值注册成 Bean。
> 如果 `@Bean` 方法本身有参数，那么这些参数会由 Spring 自动注入，所以它常被放到方法注入里一起讨论。
> Spring 依赖注入底层大致是这样的：先由注解相关的后置处理器识别注入点，再把注入点封装成依赖描述符，最后交给 BeanFactory 的 `resolveDependency` 去容器里解析并完成注入。
