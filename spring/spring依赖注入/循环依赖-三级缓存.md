# Spring 使用三级缓存解决循环依赖

## 1. 什么是循环依赖

循环依赖，指的是两个或多个 Bean 之间互相持有对方的引用，形成闭环。最常见的例子就是 A 依赖 B，同时 B 又依赖 A。这里说的“循环依赖”是对象之间的依赖关系，不是方法之间互相调用。([腾讯云][2])

```java
@Service
public class A {
    @Autowired
    private B b;
}

@Service
public class B {
    @Autowired
    private A a;
}
```

在普通 `new` 对象的场景下，如果构造过程形成这种闭环，很容易一路递归创建下去，最终触发栈溢出；而在 Spring 中，针对一部分场景，容器会通过“提前暴露引用”的方式把问题兜住。([腾讯云][2])

---

## 2. Spring 里循环依赖并不是都能解决

### 2.1 构造器注入的循环依赖，不能解决

如果 A 的构造器需要 B，B 的构造器又需要 A，那么 Spring 会直接抛出 `BeanCurrentlyInCreationException`。原因很直接：Spring 解决循环依赖依赖的是“中间态 Bean”，也就是**已经实例化，但还没有完成属性填充和初始化**的对象；而构造器注入的问题发生在“实例化之前或实例化当中”，此时连半成品对象都拿不到。([腾讯云][2])

```java
@Service
public class A {
    public A(B b) {
    }
}

@Service
public class B {
    public B(A a) {
    }
}
```

---

### 2.2 字段注入或 setter 注入的循环依赖，单例情况下通常可以解决

如果是字段注入，或者本质相同的 setter 注入，Spring 通常可以解决。因为这类依赖发生在 Bean **实例化之后、属性填充阶段**，此时对象本体已经创建出来了，容器可以先把这个“还没填完属性”的对象提前暴露出去，再让别的 Bean 先拿去引用。([腾讯云][2])

```java
@Service
public class A {
    @Autowired
    private B b;
}

@Service
public class B {
    @Autowired
    private A a;
}
```

---

### 2.3 prototype 原型作用域的循环依赖，不能靠这套机制解决

对于 `prototype` Bean，Spring 默认不会像单例那样维护完整的单例缓存体系。实际使用到时，如果原型 Bean 之间形成字段注入的循环依赖，仍然会抛出异常。文章里通常会强调一点：**三级缓存主要解决的是单例 Bean 的属性循环依赖，不解决 prototype 的循环依赖。**([掘金][1])

```java
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
@Service
public class A {
    @Autowired
    private B b;
}

@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
@Service
public class B {
    @Autowired
    private A a;
}
```

另外，给依赖加 `@Lazy` 只能延迟初始化，不能从根本上消除原型循环依赖；真正触发创建时，问题依旧可能暴露。([腾讯云][2])

---

## 3. 先看懂 Bean 的创建流程，才能看懂三级缓存

理解三级缓存之前，要先知道 Spring 创建 Bean 时的大致步骤：

1. `createBeanInstance`：实例化，底层就是调用构造器，先把对象创建出来。
2. `populateBean`：属性填充，也就是处理 `@Autowired`、setter 注入等。
3. `initializeBean`：初始化，执行各种回调、后置处理器逻辑，例如初始化方法、AOP 代理包装等。([腾讯云][2])

循环依赖通常发生在第二步，也就是 `populateBean` 阶段。因为这时 A 自己已经“出生”了，但它的属性还没填完；当 Spring 发现 A 依赖 B，就去创建 B；B 在填充属性时又发现依赖 A，这时如果能拿到 A 的“早期引用”，循环就有机会被打破。([腾讯云][2])

---

## 4. Spring 的三级缓存分别是什么

Spring 在 `DefaultSingletonBeanRegistry` 中维护了几组关键缓存。与循环依赖直接相关的就是下面这三层：([腾讯云][2])

```java
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256); // 一级缓存
private final Map<String, Object> earlySingletonObjects = new HashMap<>(16);       // 二级缓存
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16); // 三级缓存
```

### 4.1 一级缓存：`singletonObjects`

一级缓存存放的是**完全初始化好的单例 Bean**。从这里取出来的对象，可以直接使用。这也是平时通过 Spring 容器拿到的大多数 Bean 的最终落点。([腾讯云][2])

### 4.2 二级缓存：`earlySingletonObjects`

二级缓存存放的是**提前曝光的单例对象**，也可以理解成“早期对象”或“半成品对象”。它们已经实例化出来了，但还没有完成所有属性填充和初始化。它的作用是：在循环依赖发生时，让其他 Bean 可以先拿到这个早期引用。([腾讯云][2])

### 4.3 三级缓存：`singletonFactories`

三级缓存存放的不是 Bean 本身，而是 **`ObjectFactory` 工厂对象**。当别的 Bean 需要这个正在创建中的 Bean 时，Spring 会调用工厂的 `getObject()` 方法，按需拿到一个早期引用。这个设计很关键，因为它不仅能暴露原始对象，还能在必要时提前暴露代理对象。([腾讯云][2])

---

## 5. Spring 取单例时，是怎么查这三层缓存的

Spring 获取单例对象时，大致是这样的顺序：([腾讯云][2])

1. 先查一级缓存 `singletonObjects`。
2. 如果没查到，并且这个 Bean 正在创建中，再查二级缓存 `earlySingletonObjects`。
3. 如果还没有，并且允许早期引用，就去三级缓存 `singletonFactories` 里拿到 `ObjectFactory`，调用 `getObject()` 创建早期引用。创建成功后，这个早期引用会被放入二级缓存，同时从三级缓存移除。([腾讯云][2])

这一步特别重要：**对象不是直接长期放在二级缓存里等着别人取，而是先放一个工厂到三级缓存，等真的有人需要时，再由工厂产出早期引用，然后移到二级缓存。**([腾讯云][2])

---

## 6. 结合 A 依赖 B、B 依赖 A，看整个解决过程

下面用最典型的单例字段注入场景，把流程从头走一遍。这个过程也是多数文章讲三级缓存时的主线。([博客园][3])

### 第一步：创建 A

Spring 先尝试获取 A。一级、二级、三级缓存里都没有，于是开始真正创建 A。首先执行实例化，也就是 `createBeanInstance`，此时 A 对象已经被构造出来，但它的属性还没填。([博客园][3])

接着，Spring 会判断当前 Bean 是否需要“提前暴露”。如果 A 是单例、允许循环依赖，并且当前确实处于创建中，那么 Spring 会把一个能够返回 A 早期引用的 `ObjectFactory` 放进三级缓存。源码层面表现为 `addSingletonFactory(beanName, () -> getEarlyBeanReference(...))`。([腾讯云][2])

### 第二步：A 填充属性时，发现依赖 B

进入 `populateBean` 阶段，Spring 发现 A 需要注入 B，于是开始创建 B。此时 B 也和 A 一样：先实例化，再把自己的 `ObjectFactory` 提前放入三级缓存。([博客园][3])

### 第三步：B 填充属性时，发现依赖 A

轮到 B 做属性填充时，发现自己又依赖 A。此时 Spring 再去取 A：一级缓存没有，因为 A 还没初始化完成；二级缓存也可能没有，因为 A 的早期对象还没真正生成；于是 Spring 到三级缓存里找到 A 对应的 `ObjectFactory`。([腾讯云][2])

这时，Spring 调用工厂的 `getObject()`，获得 A 的一个早期引用。获得后，A 会从“三级缓存中的工厂”转为“二级缓存中的早期对象”，同时三级缓存中的那份工厂被删除。B 于是成功拿到了 A 的引用，自己的属性填充可以继续往下走。([腾讯云][2])

### 第四步：B 完成创建，进入一级缓存

B 的属性已经填好，初始化也执行完，B 就从“创建中”状态变为一个完整 Bean，最后放入一级缓存。此时，B 已经是一个可直接使用的完整对象。([博客园][3])

### 第五步：回到 A，继续完成创建

A 此前只是卡在“需要 B”这一步。现在 B 已经完整创建好了，A 就可以把 B 注入进去，完成自己的属性填充和初始化。完成后，A 也进入一级缓存。到这里，A 和 B 都成功创建完毕，循环依赖被打通。([博客园][3])

如果只看这个过程，可以把它概括成一句话：**A 还没完全创建完时，先把自己的“可引用版本”提前暴露出去，让 B 能先拿到；等 B 创建完成后，再回过头把 A 补齐。**([博客园][3])

---

## 7. 为什么一定是三级缓存，不是二级缓存

这是这个主题里最关键、也最容易被问到的点。答案不在于“多一层更保险”，而在于：**第三层缓存里放的是工厂，不是对象本身；工厂可以决定暴露出来的到底是原始对象，还是代理对象。**([腾讯云][2])

如果没有 AOP、事务这些代理需求，二级缓存看起来好像也能勉强工作：只要把实例化后的半成品对象直接塞进去，后面谁需要就拿走。但一旦 Bean 需要被代理，问题就变了：容器最终真正要放到一级缓存、要提供给业务代码使用的，可能不是原始对象，而是代理对象。([腾讯云][2])

Spring 在创建 Bean 时，会通过 `getEarlyBeanReference` 这个扩展点，让 `SmartInstantiationAwareBeanPostProcessor` 有机会提前返回“早期引用”；如果自动代理创建器介入，这个早期引用就可能直接是一个代理对象。也就是说，三级缓存里的 `ObjectFactory` 不是单纯为了“延迟一下再创建”，而是为了在**真正有人依赖我时，动态决定暴露哪种引用**。([腾讯云][2])

如果只有二级缓存，往里放的就只能是某个已经确定好的对象。一旦放早了，后面再发现这个 Bean 其实应该被代理，就会出现引用不一致的问题：某些地方拿到的是原始对象，某些地方拿到的是代理对象。这样 AOP、事务之类的增强就可能失效，或者行为变得不一致。三级缓存通过 `ObjectFactory + getEarlyBeanReference`，把这件事处理成了“按需生成一致的早期引用”，这才是它存在的真正价值。([腾讯云][2])

---

## 8. `singletonsCurrentlyInCreation` 这类标记也很重要

除了三级缓存，Spring 还会维护一个“当前正在创建中的 Bean 集合”。一个 Bean 开始创建时，会被标记进入这个集合；创建结束后再移除。这样，Spring 在取单例时就能判断：这个 Bean 是“压根没有”，还是“正在创建中，可以尝试走早期引用逻辑”。([腾讯云][2])

这也是为什么 `getSingleton(beanName, true)` 里的判断条件很关键：如果一个 Bean 不在“当前创建中”集合里，Spring 不会贸然从三级缓存去取早期引用；只有确认它确实是创建到一半，才允许走后续的提前暴露逻辑。([腾讯云][2])

---

## 9. 为什么构造器循环依赖不行，再说透一点

很多人第一次听到“Spring 能解决循环依赖”时，会误以为任何形式都能解决。其实不是。Spring 解决的是**单例 Bean 在属性注入阶段发生的循环依赖**，不是任意形态的互相依赖。([掘金][1])

构造器注入的问题在于：A 要想被创建出来，先得有 B；而 B 要想创建出来，又得先有 A。这里没有任何一个对象能先完成“实例化”这一步，也就没有所谓的半成品对象可供提前暴露。因此三级缓存根本没有介入机会。([腾讯云][2])

而字段注入或 setter 注入不同：对象可以先通过无参构造器或普通构造器实例化出来，等进入 `populateBean` 再处理依赖。这就给了 Spring 一个中间缓冲区，让它可以把“尚未完成初始化的对象引用”先拿出来顶住。([腾讯云][2])

---

## 10. 非单例 Bean 为什么也不行

原型 Bean 的语义是“每次获取都创建一个新对象”，它不走单例缓存那套完整的共享逻辑。也正因为如此，Spring 没办法像处理单例那样，把正在创建中的某个对象稳定地提前暴露给另一个依赖方，并最终保证整个容器里引用的是同一份受控对象。([腾讯云][2])

所以，文章里通常会把结论收束成一句话：**Spring 三级缓存解决的是单例模式下的属性循环依赖；对于构造器循环依赖、prototype 循环依赖，不适用。**([掘金][1])

---

## 11. 把整个机制压缩成一段适合记忆的话

Spring 创建单例 Bean 时，先实例化对象；如果这个 Bean 可能会被别的 Bean 在创建过程中依赖到，Spring 会先把一个能返回“早期引用”的工厂放进三级缓存。等到真的发生循环依赖，容器再通过这个工厂拿到早期引用，把它转移到二级缓存供其他 Bean 注入。等 Bean 完成属性填充、初始化、可能的代理包装后，再进入一级缓存，成为最终可用的完整 Bean。([腾讯云][2])

---

## 12. 面试或复习时可以直接这样回答

可以先给出结论：Spring 主要通过**三级缓存**解决**单例 Bean 的字段注入或 setter 注入的循环依赖**。三级缓存分别是：

* 一级缓存：`singletonObjects`，存放完整 Bean。
* 二级缓存：`earlySingletonObjects`，存放早期曝光对象。
* 三级缓存：`singletonFactories`，存放 `ObjectFactory`，用于按需生成早期引用。([腾讯云][2])

再补一句核心原因：之所以不是二级缓存，而要三级缓存，是因为 Spring 需要借助 `ObjectFactory` 和 `getEarlyBeanReference` 在出现循环依赖时决定暴露原始对象还是代理对象，从而保证 AOP、事务等场景下引用一致。([腾讯云][2])

最后再补边界条件：构造器循环依赖解决不了，prototype 循环依赖也解决不了。([腾讯云][2])

---

## 13. 朗读版结尾总结

Spring 之所以能处理一部分循环依赖，不是因为它“无所不能”，而是因为它抓住了 Bean 创建过程中的一个窗口期：对象已经实例化，但还没完成属性填充和初始化。借助这个窗口期，Spring 通过三级缓存提前暴露引用，先把依赖链接上，再回头完成对象创建。而三级缓存里最关键的一层，不是存对象的二级缓存，而是存工厂的三级缓存，因为它决定了早期暴露出来的引用能否与最终对象保持一致，尤其是在 AOP 和代理场景下。([腾讯云][2])

# 问题：代理对象是什么时候出现的？
这里的**代理对象**，是在 **Bean 创建过程中由 Spring 的后置处理器决定并生成的**，不是一开始 `new` 出来时就有。

你可以按时间顺序这样理解：

---

## 1. 最开始出现的是原始对象

Spring 创建 Bean 的第一步是实例化：

```java
A a = new A();
```

这时候拿到的只是**原始对象**，还没有依赖注入，也没有 AOP 代理。

---

## 2. 真正可能出现代理对象的阶段

代理对象通常出现在 **BeanPostProcessor** 参与处理的时候，尤其是和 AOP 相关的后置处理器，比如：

* `AbstractAutoProxyCreator`
* 事务相关的代理创建器
* AOP 切面相关处理器

这些处理器会判断：

* 这个 Bean 是否需要增强
* 是否要织入事务、切面、拦截器
* 如果需要，就返回一个代理对象，而不是原始对象

---

## 3. 正常情况下，代理对象通常在初始化阶段后出现

标准流程大致是：

1. 实例化原始对象
2. 属性注入
3. 初始化
4. `BeanPostProcessor` 前后处理
5. 如果需要增强，返回代理对象

也就是说，**最终放进一级缓存的，可能已经不是原始对象，而是代理对象**。

---

## 4. 那为什么三级缓存又说“提前暴露代理对象”

因为循环依赖时，不能等整个初始化都做完。

例如：

* A 依赖 B
* B 依赖 A
* 并且 A 还需要被 AOP 代理

这时如果 B 注入的是 **原始 A**
而最后容器里放进去的是 **代理 A**
就会出现不一致：

* B 手里拿的是原始对象
* 其他地方拿的是代理对象

这样 AOP、事务可能就失效了。

所以 Spring 在循环依赖场景下，会在**还没完全初始化结束之前**，通过三级缓存里的 `ObjectFactory` 调用：

```java
getEarlyBeanReference(...)
```

这个方法会让后置处理器提前介入，看看：

> 这个 Bean 将来是不是要代理？
> 如果要，那现在提前暴露的时候就直接给一个代理引用。

所以：

* **没有循环依赖时**：代理对象多半在初始化后，由后置处理器包出来
* **有循环依赖时**：为了保证注入一致，Spring 可能在“提前暴露”阶段就先生成一个早期代理对象

---

## 5. 关键点：谁创建了代理对象

不是三级缓存自己创建代理。

三级缓存里放的是：

```java
ObjectFactory
```

它的作用是：**当别人真的需要这个早期引用时，再调用工厂方法去拿对象**。

而这个工厂内部通常会调用：

```java
getEarlyBeanReference(beanName, mbd, bean)
```

在这里，AOP 相关的后置处理器有机会把原始对象包装成代理对象。

所以更准确地说：

* **代理对象是由 BeanPostProcessor / AutoProxyCreator 这类机制生成的**
* **三级缓存只是给了它一个“提前生成并暴露”的时机**

---

## 6. 你可以把它记成两种时机

### 情况一：没有循环依赖

* 先完整创建原始对象
* 初始化后
* 后置处理器判断要不要代理
* 最终生成代理对象

### 情况二：有循环依赖

* 原始对象刚实例化出来
* 还没完全初始化
* 为了让别的 Bean 能注入它
* Spring 通过三级缓存提前拿“早期引用”
* 如果判断它将来需要代理，就在这一步提前返回代理对象

---

## 7. 一句话回答你的问题

**这里说的代理对象，本质上是在 Spring 的后置处理器执行时生成的；在循环依赖场景下，为了保证依赖注入拿到的对象和最终容器中的对象一致，Spring 会借助三级缓存把这个代理对象提前暴露出来。**

---

## 8. 最容易懂的总结

你可以简单理解成：

* 刚 `new` 出来的是**原始对象**
* 经过 AOP 等处理后，可能变成**代理对象**
* 如果存在循环依赖，Spring 不能等最后再代理
* 否则别人注入进去的就会是“旧的原始对象”
* 所以 Spring 会在三级缓存阶段，**提前把将来的代理对象拿出来给别人用**
